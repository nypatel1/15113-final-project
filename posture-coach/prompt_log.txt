AI Posture Coach — prompt / development log
============================================

This is an intent-level record of how the project was built, not a
verbatim chat transcript. Each entry is the gist of a prompt I gave
to a Cursor cloud agent (Claude, same agentic pattern as HW8), the
resulting PR, and any follow-ups from me after reviewing the diff.

Workflow: each change got its own feature branch → commit → push →
PR → CI (lint + node --test + build) → I merged on GitHub. That kept
the unit of work at the PR level so bad ideas could be reverted with
one click.

------------------------------------------------------------
Pre-PR — Scoping the build
------------------------------------------------------------
Used Claude (browser chat) to turn the open-ended assignment spec
into SPEC.md: functional requirements, data flow, guiding principles.
No code yet. The goal was to have a document I could point the
agent at so every later prompt could be "narrow, scoped, one thing".

------------------------------------------------------------
PR #1 — Initial build (merged)
------------------------------------------------------------
Prompt: "Read SPEC.md. Scaffold a Vite + React 19 app under
posture-coach/. Build it in modular slices:

  1. Core posture analysis engine as a dependency-free JS module with
     unit tests against synthetic landmark data (no DOM required).
  2. A MediaPipe Pose Landmarker wrapper so the rest of the app never
     has to care which model is underneath.
  3. The React UI: webcam + canvas overlay + metric bars + tips.
  4. localStorage persistence with a capped session schema.
  5. History tab with Recharts line and bar views + CSV export.

Pick 'lite' float16 model (fast startup, enough quality for upper-body
posture). Use a smooth exp(-t^2) penalty curve for sub-scores so the
score doesn't feel binary. Renormalise weights when a metric is null
so seated users (hips off-camera) still get a 0-100 scale. Commit
each slice separately."

What landed: 6 commits (spec + README, scaffold, analysis engine +
tests, detector + store, UI, docs). 7/7 tests pass.

Review notes I caught on the diff:
- Mirror the video horizontally (transform: scaleX(-1)) so left/right
  matches the user's body.
- Sample into sessions at 1 Hz, not every frame — 60 Hz metric arrays
  would blow localStorage inside an hour.
- PoseLandmarker in VIDEO mode needs a monotonic timestamp
  (performance.now()), or it throws.

PR #4 — Head pitch metric (merged)
------------------------------------------------------------
Bug I found while using the live site: the score chip stayed yellow
when I sat upright and turned *green* when I tucked my chin to my
chest. Asked the agent to diagnose before fixing.

Diagnosis: the existing `neckTilt` metric only measures ears-vs-
shoulders (forward-head drift). When you nod your chin to your chest
your head pivots at the neck — the ears barely move, sometimes drift
slightly back — so the metric read "less forward head = better", the
opposite of the truth.

Prompt: "Add a headPitch metric using the nose and mouth face
landmarks. Definition: (nose.y - earMidpoint.y) / |leftEar - rightEar|.
~0.3 = level gaze, larger = chin tucked down, smaller/negative = chin
lifted. Independent of neckTilt by construction (whole-head
translation moves ears and nose together). Fall back to mouth
midpoint when the nose is occluded. Add it to buildBaseline,
scorePosture, the feedback rules, the FeedbackPanel row, the
HistoryView bar chart, and the PoseCoach EMA smoother. Write tests
that (a) assert chin-tuck lowers the sub-score with a 'tucked'
message, (b) chin-lift produces a 'lifted' message, (c) headPitch is
stable under forward-head-only translation and neckTilt is stable
under chin-tuck-only translation (independence test)."

Went from 7/7 tests → 10/10.

------------------------------------------------------------
Two reports at once:

  (a) "If I'm leaning super far ahead the score can be green, but
      when I sit back with good posture it goes yellow."
  (b) "Can you draw lines on the chin or frame the face?"

Prompt: "Fix (a) by adding a torsoLean metric from the z-coordinate
of the shoulders (MediaPipe z grows toward the camera), normalised by
shoulder width. Also switch shoulderHunch to one-sided (only penalise

the ear line (rounded rectangle, dashed ear horizon, nose dot, chin
line extending from ear midpoint through nose to a perpendicular
tick). Everything scales by ear distance. Add tests for both."

14/14 tests passed. Shipped it.

Result in practice: felt worse to use. User said "it's bad now,
revert". Honest lesson: this PR bundled three unrelated changes
(new metric + penalty change + UI overlay). If any one of them was
wrong, the whole thing had to go. Smaller PRs from this point on.

PR #7 — One-sided shoulder hunch (open, draft)
------------------------------------------------------------
Report: "The shoulder hunch doesn't work well, I have to squeeze my
shoulders very high for it to improve."

Root cause: gradedScore takes Math.abs(deviation), so a longer-than-
baseline neck was penalised just as hard as a shorter one. Longer
neck actually means *relaxed shoulders*, which is what we want; the
only way to satisfy the metric was to shrug up.

Prompt: "Add a gradedScoreOneSided(deviation, tolerance, falloff,
sign) helper. Switch `hunch` to use it with sign=-1 so only neck
compression loses points. Simplify the hunch feedback message (the
'back is overextended' branch can no longer fire). Do not touch any
other metric. Add tests: relaxed shoulders (longer neck) stays at
100 with no feedback; shrugged shoulders (shorter neck) still drops
the sub-score and produces the 'creeping up' message."

Surgical: touches only postureAnalysis.js and its test file.
12/12 tests.

------------------------------------------------------------
PR #8 — Asymmetric head-pitch tolerance (open, draft)
------------------------------------------------------------
Report: "The head positioning is a little sensitive up-down. I should
be able to still move my head up a little and it should be green."

Root cause: the `head` sub-score used a single symmetric tolerance
for both chin-up and chin-down. But chin-down (tucked toward chest)
is the real ergonomic problem, while chin-up a little usually just
means the monitor is at eye-level. The penalty shouldn't be the
same size on both sides.

Prompt: "Add a gradedScoreAsymmetric(deviation, posTolerance,
posFalloff, negTolerance, negFalloff) helper. Use it for the `head`
sub-score with (0.05, 0.25, 0.12, 0.40) — unchanged strict
penalty for chin-down, 2.4x wider dead zone for chin-up with a
gentler falloff. Add three tests: small chin lift keeps sub-score at
100, extreme chin lift still flagged, and same magnitude of chin-up
vs chin-down scores higher for chin-up."

------------------------------------------------------------
Meta-strategies that worked
------------------------------------------------------------
1. Spec before code. Having SPEC.md let every later prompt be a
   narrow "change one thing" instead of "build the app".
2. Analysis engine before UI. Made the interesting math testable in
   isolation — no webcam required to verify scoring.
3. AI as reviewer, not just typist. "What's wrong with this approach"
   and "explain the bug before you fix it" produced much better
   results than "add feature X". PR #4 started as a diagnosis
   request, not a feature request.
4. Small PRs with scope statements. Every post-#1 prompt explicitly
   said "don't touch anything else". When I broke that rule (PR #5),
   I shipped a bad change I couldn't tune — only revert.
5. Tests as regression pins. Every bug-fix PR after #4 added a test
   that would have caught the bug. That grew the suite from 7 → 10
   → 12 entries organically.
